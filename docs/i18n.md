# UI i18n — the bilingual inspector chrome

The web inspector's chrome is bilingual (de/en). The active UI language is resolved
per request — `?lang=` → `ui_lang` cookie → persisted setting — independent of the
generated CONTENT language (which follows the user's input language and is never
switched by the UI toggle).

- **Catalog**: `sonaloop/web/_i18n.py` `STRINGS` — every key exists in every
  supported language with the same `{placeholder}`s (`tests/test_i18n.py`).
- **Switcher**: the sidebar settings popover (bottom-left) carries the de/en
  toggle; an explicit choice persists via the cookie + `set_ui_language()`.
- **Leak guard**: `tests/test_i18n_leaks.py` lint-scans the web package — a
  hard-coded button label / page title / empty state fails CI. Long doc prose in
  the docs hub uses the inline `("…" if de else "…")` pair pattern instead of
  catalog keys (bilingual by construction; see `web/_docs.py`).

## Extensions: registering translated strings

Downstream packages (sonaloop-cloud / sonaloop-research) ship their own UI
strings without touching the core catalog. From the same `setup(app)` entry point
that registers routes/nav (`sonaloop.web.extensions`):

```python
from sonaloop.web._i18n import register_strings

def setup(app):
    register_strings("en", {"cloud.usage_h": "Usage", "cloud.runs_n": "{n} runs"})
    register_strings("de", {"cloud.usage_h": "Verbrauch", "cloud.runs_n": "{n} Runs"})
    # … routes / nav items resolve labels with lambda: t("cloud.usage_h")
```

Rules (enforced by `register_strings` + `extension_parity_problems()`):

- keys MUST be namespaced — `"<ns>.<key>"`, e.g. `cloud.usage_h`. Core keys are
  flat, so an extension can never shadow or override one (a flat key raises).
- register for **every** supported language; `extension_parity_problems()`
  returns the gaps (missing languages, `{placeholder}` mismatches) — assert it is
  empty in the extension's own test suite.
- resolve with the normal `t("cloud.usage_h")` — fallback behaviour is identical
  to core keys.
